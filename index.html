📖 Introduction à Kubernetes

            

                Kubernetes (souvent abrégé K8s) est un orchestrateurd de conteneurs open-source, initialement développé par Google. 
                Il permet d’automatiser le déploiement, la mise à l’échelle et la gestion d’applications conteneurisées.
            

            

                Pourquoi utiliser Kubernetes ?

                - Résilience : redémarre automatiquement les conteneurs défaillants, remplace les nœuds défectueux.

                - Scalabilité : augmente ou diminue le nombre de réplicas en fonction de la charge (autoscaling).

                - Portabilité : fonctionne sur n’importe quel cloud (AWS, GCP, Azure) ou en on‑premise.

                - Déploiements déclaratifs : vous décrivez l’état souhaité (ex : 3 réplicas de Nginx) et Kubernetes s’assure que cet état est atteint.
            

            

                Comment ça marche ? Un cluster Kubernetes est composé d’un plan de contrôle (API Server, etcd, Scheduler, Controller Manager) et de nœuds de travail (workers) qui exécutent les conteneurs.
            


            
            
🌐 Comment fonctionne le réseau dans Kubernetes

            

                Le réseau est un des aspects les plus importants (et parfois complexes) de Kubernetes. Voici les principes fondamentaux :
            

            
    
                
  • Chaque Pod reçoit une adresse IP unique (et non chaque conteneur). Les Pods peuvent communiquer entre eux sans NAT (Network Address Translation).
    • 
                
  • Le kube-proxy est un composant qui gère les règles de réacheminement (iptables, IPVS) sur chaque nœud. Il assure que les requêtes vers un Service (ClusterIP) soient distribuées aux Pods correspondants (load‑balancing interne).
    • 
                
  • Le CNI (Container Network Interface) est le plugin qui attribue les adresses IP aux Pods et gère la connectivité entre les nœuds. Les CNI populaires : Calico (supporte les Network Policies), Flannel (simple), Cilium (eBPF), Weave.
    • 
                
  • Les Services créent une abstraction stable (ClusterIP) par‑dessus les Pods éphémères. Ils peuvent être de type ClusterIP (interne), NodePort (exposé sur un port de chaque nœud) ou LoadBalancer (IP externe provisionnée par MetalLB ou le cloud).
    • 
                
  • L’Ingress apporte une couche supplémentaire de routage HTTP/HTTPS, permettant de rediriger le trafic vers différents Services selon le nom d’hôte ou le chemin.
    • 
            

            

                Résumé du flux réseau : Utilisateur → LoadBalancer (externe) → Ingress (optionnel) → Service (ClusterIP) → Pod.
            


            
            
🧩 Concepts clés

            

                

                    
Pod

                    
Définition : Un Pod est le plus petit objet déployable. Il regroupe un ou plusieurs conteneurs qui partagent le même réseau (une seule IP) et le même espace de stockage (volumes).

                    
Exemple YAML :

                    
apiVersion: v1
kind: Pod
metadata:
  name: mon-pod
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80

                

                

                    
Service

                    
Définition : Un Service expose un ensemble de Pods (sélectionnés par des labels) via une adresse IP et un port stables. Types : ClusterIP (interne), NodePort (externe sur un port statique), LoadBalancer (IP externe).

                    
apiVersion: v1
kind: Service
metadata:
  name: mon-service
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP

                

                

                    
Ingress

                    
Définition : Un Ingress est un objet qui expose des services HTTP/HTTPS à l'extérieur du cluster en utilisant un nom d'hôte ou un chemin d'URL. Il agit comme un routeur (reverse proxy).

                    
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: mon-ingress
spec:
  rules:
  - host: monapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: mon-service
            port:
              number: 80

                

                

                    
Deployment

                    
Définition : Un Deployment est un contrôleur qui gère un ensemble de Pods (via un ReplicaSet) en offrant des mises à jour déclaratives (rolling update, rollback, scaling).

                    
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21

                

                

                    
ConfigMap

                    
Définition : Stocke des données de configuration non sensibles (ex : variables d’environnement, fichiers de configuration). Permet de découpler la configuration du code.

                    
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  APP_DEBUG: "false"

                

                

                    
Secret

                    
Définition : Similaire à un ConfigMap, mais pour des données sensibles (mots de passe, tokens, clés SSH). Les données sont encodées en base64 et peuvent être chiffrées au repos.

                    
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
data:
  password: dXNlcg==    # base64 de "user"

                

            


            
            
🛠️ Commandes kubectl essentielles

            

                
🔍 kubectl cluster-info

                
Affiche l'URL du serveur API.

                
$ kubectl cluster-info
Kubernetes control plane is running at https://172.18.0.2:6443

                💡 Vérifie que vous êtes bien connecté au cluster.
            

            

                
📋 kubectl get nodes

                
Liste les nœuds avec leur statut.

                
$ kubectl get nodes
NAME                 STATUS   ROLES           AGE
kind-control-plane   Ready    control-plane   5m
kind-worker          Ready    <none>         5m

            

            

                
🔎 kubectl describe pod <nom>

                
Détails et événements d'un Pod (indispensable pour le debug).

                💡 La section Events est précieuse pour comprendre pourquoi un Pod ne démarre pas.
            

            

                
📄 kubectl apply -f fichier.yml

                
Crée ou met à jour des ressources à partir d'un fichier YAML (déclaratif).

            

            

                
📜 kubectl logs <pod>

                
Affiche les logs du conteneur.

            

            

                
🖥️ kubectl exec -it <pod> -- /bin/bash

                
Ouvre un shell interactif dans le conteneur pour diagnostiquer.

            


            
            
🔒 Sécurité & Administration (bonnes pratiques)

            

                Objectif : Garantir la confidentialité, l'intégrité et la disponibilité du cluster.
            

            
    
                
  • RBAC (Role-Based Access Control) : définit qui peut faire quoi sur quelles ressources.
    • 
                
  • Network Policies : isole le trafic entre Pods (ex : seule l'application front-end peut accéder à la base de données).
    • 
                
  • Pod Security Standards : profils Privileged, Baseline, Restricted.
    • 
                
  • Gestion des Secrets : chiffrement au repos dans etcd, utilisation d'outils comme External Secrets Operator.
    • 
                
  • Resource Quotas : limite la consommation CPU/mémoire par namespace.
    • 
                
  • Security Context : exécute les conteneurs avec un utilisateur non-root et sans capacités Linux superflues.
    • 
                
  • Audit Logs : enregistre toutes les requêtes vers l'API Server.
    • 
                
  • CIS Benchmarks : utilisez kube-bench pour auditer votre cluster.
    •

Exemple 1 : Déployer Nginx avec un service LoadBalancer

            

                
🎯 Objectif : Exposer une application Web Nginx à l'extérieur du cluster via une IP publique (ou locale avec MetalLB).

                
📄 Étape 1 : Créer le fichier nginx-lb.yml

                
# nginx-lb.yml
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
  type: LoadBalancer

                
🖥️ Étape 2 : Appliquer

                
kubectl apply -f nginx-lb.yml

                
deployment.apps/nginx-deploy created
service/nginx-service created

                
🔍 Étape 3 : Vérifier les Pods

                
kubectl get pods

                
NAME                            READY   STATUS    RESTARTS   AGE
nginx-deploy-7d8f8-5xg7f        1/1     Running   0          30s
nginx-deploy-7d8f8-6h8g9        1/1     Running   0          30s

                
🌍 Étape 4 : Récupérer l'IP externe

                
kubectl get svc nginx-service

                
NAME            TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)
nginx-service   LoadBalancer   10.96.123.45    192.168.1.240    80:30001/TCP

                
🧪 Étape 5 : Tester

                
curl http://192.168.1.240

                
<!DOCTYPE html>... (page Nginx)

                
🗑️ Étape 6 : Nettoyer

                
kubectl delete -f nginx-lb.yml

            


            
            
Exemple 2 : Utiliser un ConfigMap

            

                
🎯 Objectif : Injecter des variables d'environnement via ConfigMap.

                
📄 Étape 1 : ConfigMap

                
# app-config.yml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  APP_DEBUG: "false"

                
📄 Étape 2 : Pod utilisant le ConfigMap

                
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: busybox
    command: ["/bin/sh"]
    args: ["-c", "echo $APP_ENV; sleep 3600"]
    envFrom:
    - configMapRef:
        name: app-config

                
🖥️ Étape 3 : Appliquer et tester

                
kubectl apply -f app-config.yml
kubectl apply -f pod-with-config.yml
kubectl logs app-pod

                
production

            


            
            
Exemple 3 : Ingress multi-hôtes

            

                
🎯 Objectif : Router deux applications sur deux noms de domaine différents.

                
📄 Fichier ingress-example.yml

                
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: mon-ingress
spec:
  rules:
  - host: app1.mondomaine.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service-app1
            port:
              number: 80
  - host: app2.mondomaine.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service-app2
            port:
              number: 80

                
🖥️ Appliquer et vérifier

                
kubectl apply -f ingress-example.yml
kubectl get ingress

                
NAME          CLASS    HOSTS                                ADDRESS
mon-ingress   <none>   app1.mondomaine.com,app2.mondomaine.com   192.168.1.240

            


            
            
Exemple 4 : Stockage persistant (PVC)

            

                
🎯 Objectif : Conserver des données même après redémarrage du Pod.

                
📄 PVC et Pod

                
# pvc.yml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mon-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
---
# pod-avec-stockage.yml
apiVersion: v1
kind: Pod
metadata:
  name: pod-avec-stockage
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: stockage
      mountPath: /usr/share/nginx/html
  volumes:
  - name: stockage
    persistentVolumeClaim:
      claimName: mon-pvc

                
🖥️ Appliquer et tester la persistance

                
kubectl apply -f pvc.yml
kubectl apply -f pod-avec-stockage.yml
kubectl exec pod-avec-stockage -- sh -c "echo 'Persistant' > /usr/share/nginx/html/test.html"
kubectl delete pod pod-avec-stockage
kubectl apply -f pod-avec-stockage.yml
kubectl exec pod-avec-stockage -- cat /usr/share/nginx/html/test.html

                
Persistant

                🎉 Les données sont bien persistantes !
