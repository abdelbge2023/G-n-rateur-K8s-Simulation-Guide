📖 Introduction à Kubernetes

            

                Kubernetes (souvent abrégé K8s) est un orchestrateur de conteneurs open-source, initialement développé par Google. 
                Il permet d’automatiser le déploiement, la mise à l’échelle et la gestion d’applications conteneurisées.
            

            

                Objectifs de la formation :

                - Comprendre les cas d'utilisation des Pods.

                - Appliquer des exemples d'architectures.

                - Définir le rôle des services.
            


            
            
🌐 Comment fonctionne le réseau dans Kubernetes

            
    
                
  • Chaque Pod reçoit une adresse IP unique (et non chaque conteneur). Les Pods peuvent communiquer entre eux sans NAT.
    • 
                
  • Le kube-proxy gère les règles de réacheminement (iptables, IPVS) pour distribuer le trafic vers les Pods.
    • 
                
  • Le CNI (Container Network Interface) attribue les adresses IP et gère la connectivité entre les nœuds.
    • 
                
  • Les Services créent une abstraction stable (ClusterIP, NodePort, LoadBalancer).
    • 
                
  • L’Ingress apporte une couche de routage HTTP/HTTPS supplémentaire.
    • 
            

            
Résumé du flux réseau : Utilisateur → LoadBalancer (externe) → Ingress (optionnel) → Service (ClusterIP) → Pod.


            
            
🧩 Les Pods et leurs caractéristiques

            

                Qu'est-ce qu'un Pod ?

                Un Pod est le plus petit objet déployable dans Kubernetes. C'est un groupe d'un ou plusieurs conteneurs qui partagent :
                
    
                    
  • Un contexte réseau (une seule adresse IP).
    • 
                    
  • Un espace de noms Linux (network, UTS, IPC, PID).
    • 
                    
  • Des cgroups pour limiter, isoler et mesurer l'utilisation des ressources.
    • 
                    
  • Des volumes partagés (stockage).
    • 
                

            

            
Deux modèles d'utilisation :

            
    
                
  • Pod avec un seul conteneur : le cas le plus courant. Le Pod est une « enveloppe » autour du conteneur.
    • 
                
  • Pod avec plusieurs conteneurs : les conteneurs doivent travailler ensemble et partager des ressources (ex: serveur LAMP, groupe DNS/NTP/DHCP).
    • 
            


            
            
🔄 Cycle de vie des Pods

            
L'objet PodStatus (champ status) reflète l'état du Pod. Voici les différentes phases :

            

                

                    
ValeurDescription

                    
                        
PendingLe Pod a été accepté par Kubernetes, mais un ou plusieurs conteneurs ne sont pas totalement créés (téléchargement des images, configuration réseau...).

                        
RunningLe Pod a été affecté à un nœud, tous les conteneurs ont été créés. Au moins un conteneur est en cours d'exécution.

                        
SucceededTous les conteneurs du Pod ont terminé avec succès et ne seront pas redémarrés (cas des Jobs).

                        
FailedTous les conteneurs ont terminé, et au moins un a terminé en échec (code non nul ou arrêté par le système).

                        
UnknownL'état du Pod ne peut pas être obtenu (généralement en cas d'erreur de communication avec l'hôte).

                        
TerminatingLe Pod est en train de terminer son exécution (grace period).

                    
                

            

            
Consulter le cycle de vie :

            
# Exporter en JSON/YAML
kubectl get pod -n <namespace> <nom-pod> -o=jsonpath='{.status.conditions}' | jq
# Variables disponibles : lastProbeTime, lastTransitionTime, message, raison, Status


            
            
🩺 Les sondes (Probes)

            
Le kubelet contrôle l'état des conteneurs via des handlers. Trois types de sondes :

            
    
                
  • ExecAction : exécute une commande dans le conteneur. Succès si code retour = 0.
    • 
                
  • TCPSocketAction : teste la connexion sur l'IP:Port du conteneur.
    • 
                
  • HTTPGetAction : effectue une requête HTTP. Succès si statut 200.
    • 
            

            
Réponse du kubelet : Success, Failure, Unknown.

            
Les trois sondes principales :

            
    
                
  • livenessProbe : vérifie si le conteneur est vivant. Échec → redémarrage.
    • 
                
  • readinessProbe : vérifie si le conteneur est prêt à recevoir du trafic. Échec → retrait des endpoints.
    • 
                
  • startupProbe : vérifie le démarrage. Désactive les autres sondes jusqu'à succès.
    • 
            

            

                
Exemple de livenessProbe avec exec

                
apiVersion: v1
kind: Pod
metadata:
  name: pod-avec-probe
spec:
  containers:
  - name: app
    image: nginx
    livenessProbe:
      exec:
        command:
        - cat
        - /tmp/healthy
      initialDelaySeconds: 5
      periodSeconds: 5

                💡 Le Pod sera redémarré si le fichier /tmp/healthy n'existe pas.
            


            
            
⚙️ Les InitContainers

            

                Définition :

                Un InitContainer est un conteneur qui s'exécute avant les conteneurs principaux du Pod. Il est conçu pour réaliser des tâches d'initialisation (attendre qu'un service soit prêt, télécharger des données, configurer l'environnement).

                Caractéristiques :
                
    
                    
  • Ils s'exécutent en série (le suivant attend la fin du précédent).
    • 
                    
  • Ils doivent se terminer avec succès pour que le Pod passe à l'étape suivante.
    • 
                    
  • En cas d'échec, le kubelet les redémarre selon restartPolicy.
    • 
                    
  • Le statut est consultable via .status.initContainerStatuses.
    • 
                

            

            

                
Exemple : Pod avec 2 InitContainers attendant des services DNS

                
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  labels:
    app.kubernetes.io/name: MyApp
spec:
  containers:
  - name: myapp-container
    image: busybox:1.28
    command: ['sh', '-c', 'echo The app is running! && sleep 3600']
  initContainers:
  - name: init-myservice
    image: busybox:1.28
    command: ['sh', '-c', 'until nslookup myservice; do echo waiting for myservice; sleep 2; done;']
  - name: init-mydb
    image: busybox:1.28
    command: ['sh', '-c', 'until nslookup mydb; do echo waiting for mydb; sleep 2; done;']

                💡 Les InitContainers attendent que les Services myservice et mydb soient résolus dans le DNS du cluster.
                
🔍 Étapes de déblocage :

                
# 1. Déployer le Pod
kubectl apply -f 04-initContainer.yml

# 2. Vérifier les logs du premier InitContainer
kubectl logs myapp-pod -c init-myservice
# Sortie : waiting for myservice - nslookup: can't resolve...

# 3. Créer le service myservice
apiVersion: v1
kind: Service
metadata:
  name: myservice
spec:
  ports:
  - protocol: TCP
    port: 80
    targetPort: 9376
# kubectl apply -f myservice.yml

# 4. Vérifier que le premier InitContainer est débloqué
kubectl logs myapp-pod -c init-myservice
# Sortie : Address 1: 10.109.138.31 myservice.kubernetes-tp.svc.cluster.local

# 5. Créer le service mydb (même principe)

# 6. Vérifier que le conteneur principal démarre
kubectl logs myapp-pod -c myapp-container
# Sortie : The app is running!

            


            
            
🛠️ Les SideCar Containers

            

                Définition :

                Un SideCar est un conteneur secondaire qui s'exécute aux côtés du conteneur principal dans le même Pod. Il étend ses fonctionnalités sans en altérer le fonctionnement.

                Cas d'usage : journalisation (log shipping), monitoring, proxy HTTPS, synchronisation de données, sécurité.

                Caractéristiques :
                
    
                    
  • Depuis Kubernetes 1.29, support natif via restartPolicy: Always sur un InitContainer.
    • 
                    
  • Ils démarrent séquentiellement mais continuent de tourner après le démarrage du Pod.
    • 
                    
  • Ils peuvent avoir leur propre readinessProbe affectant l'état global du Pod.
    • 
                

            

            

                
Exemple : Log Shipper (SideCar qui collecte les logs)

                
apiVersion: v1
kind: Pod
metadata:
  name: myapp-logs
spec:
  containers:
  - name: myapp
    image: busybox
    command: ['sh', '-c', 'while true; do echo "App log $(date)" >> /var/log/app.log; sleep 2; done']
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log
  - name: logshipper
    image: busybox
    command: ['sh', '-c', 'tail -f /var/log/app.log']
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log
  volumes:
  - name: shared-logs
    emptyDir: {}

                💡 Le conteneur logshipper lit en temps réel les logs écrits par myapp via un volume partagé.
            

            
Autres modèles de SideCar : Proxy HTTPS, synchronisation dynamique, Adaptateur, Ambassador.


            
            
🧩 Autres types de conteneurs

            
    
                
  • Ephemeral Containers : conteneurs temporaires pour débogage (ex: kubectl debug).
    • 
                
  • Ambassador : proxy inverse local vers différents environnements.
    • 
                
  • Adapter : convertit ou réorganise des données (JSON → XML, logs formatés).
    • 
            


            
            
🔧 Exemple pratique : Apache + Nginx en SideCar (proxy HTTPS)

            

                
🎯 Objectif : Exposer un serveur Apache via un proxy Nginx qui gère le HTTPS.

                
📄 Étape 1 : Déploiement Apache seul

                
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-proxy
  labels:
    app: nginx-proxy
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx-proxy
  template:
    metadata:
      labels:
        app: nginx-proxy
    spec:
      containers:
      - name: apache
        image: ubuntu/apache2
        ports:
        - containerPort: 80

                
📄 Étape 2 : Génération des certificats et création du Secret

                
openssl req -newkey rsa:2048 -nodes -keyout nginx.key -x509 -days 365 -out nginx.crt
kubectl create secret generic nginx-certs-keys --from-file=./nginx.crt --from-file=./nginx.key

                
📄 Étape 3 : Fichier de configuration Nginx (default.conf)

                
server {
    listen 443 ssl;
    server_name localhost;
    ssl_certificate /etc/nginx/ssl/nginx.crt;
    ssl_certificate_key /etc/nginx/ssl/nginx.key;
    location / {
        proxy_pass http://localhost:80;
        proxy_set_header Host $host;
    }
}

                
📄 Étape 4 : Création du ConfigMap et déploiement du Pod avec SideCar

                
kubectl create configmap nginxconfigmap --from-file=default.conf

apiVersion: v1
kind: Pod
metadata:
  name: nginx-proxy
spec:
  containers:
  - name: apache
    image: ubuntu/apache2
    ports:
    - containerPort: 80
  - name: nginxhttps
    image: nginx
    volumeMounts:
    - name: nginx-config
      mountPath: /etc/nginx/conf.d
    - name: nginx-certs
      mountPath: /etc/nginx/ssl
    ports:
    - containerPort: 443
  volumes:
  - name: nginx-config
    configMap:
      name: nginxconfigmap
  - name: nginx-certs
    secret:
      secretName: nginx-certs-keys

                
🔍 Étape 5 : Vérification

                
kubectl get pods
wget http://10.244.1.38 -O /tmp/index1
wget --no-check-certificate https://10.244.1.38 -O /tmp/index2
diff /tmp/index1 /tmp/index2   # Ne doit rien retourner

                
🌍 Étape 6 : Exposition en LoadBalancer

                
apiVersion: v1
kind: Service
metadata:
  name: nginx-proxy
spec:
  type: LoadBalancer
  ports:
  - port: 443
  selector:
    app: nginx-proxy
  externalIPs:
  - "192.168.1.140"

Exemple 1 : Déployer Nginx avec un service LoadBalancer

            

                
🎯 Objectif : Exposer Nginx à l'extérieur via une IP externe.

                
📄 Fichier nginx-lb.yml

                
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
  type: LoadBalancer

                
🖥️ Appliquer et tester

                
kubectl apply -f nginx-lb.yml
kubectl get pods
kubectl get svc nginx-service
curl http://

                
Page d'accueil Nginx

                
🗑️ Nettoyer

                
kubectl delete -f nginx-lb.yml

            


            
            
Exemple 2 : Utiliser un ConfigMap pour paramétrer une application

            

                
🎯 Objectif : Injecter des variables d'environnement via ConfigMap.

                
📄 ConfigMap (app-config.yml)

                
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  APP_DEBUG: "false"

                
📄 Pod utilisant le ConfigMap

                
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: busybox
    command: ["/bin/sh"]
    args: ["-c", "echo $APP_ENV; sleep 3600"]
    envFrom:
    - configMapRef:
        name: app-config

                
🖥️ Appliquer et vérifier

                
kubectl apply -f app-config.yml
kubectl apply -f pod-with-config.yml
kubectl logs app-pod
# Sortie : production

            


            
            
Exemple 3 : Configurer un Ingress multi‑hôtes

            

                
🎯 Objectif : Router deux applications sur deux domaines différents.

                
📄 Fichier ingress-example.yml

                
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: mon-ingress
spec:
  rules:
  - host: app1.mondomaine.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service-app1
            port:
              number: 80
  - host: app2.mondomaine.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service-app2
            port:
              number: 80

                
🖥️ Appliquer et vérifier

                
kubectl apply -f ingress-example.yml
kubectl get ingress

                💡 L'Ingress Controller (Nginx) doit être installé au préalable.
            


            
            
Exemple 4 : Stockage persistant avec PersistentVolumeClaim

            

                
🎯 Objectif : Conserver des données même après redémarrage du Pod.

                
📄 PVC et Pod

                
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mon-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: pod-avec-stockage
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: stockage
      mountPath: /usr/share/nginx/html
  volumes:
  - name: stockage
    persistentVolumeClaim:
      claimName: mon-pvc

                
🖥️ Tester la persistance

                
kubectl apply -f pvc.yml
kubectl apply -f pod-avec-stockage.yml
kubectl exec pod-avec-stockage -- sh -c "echo 'Persistant' > /usr/share/nginx/html/test.html"
kubectl delete pod pod-avec-stockage
kubectl apply -f pod-avec-stockage.yml
kubectl exec pod-avec-stockage -- cat /usr/share/nginx/html/test.html
# Sortie : Persistant

                🎉 Les données sont bien persistantes.

🔒 Sécurité & Administration (bonnes pratiques)

            

                Bonnes pratiques pour les Pods :
                
    
                    
  • Utilisez des sondes (livenessProbe, readinessProbe) pour la résilience.
    • 
                    
  • Utilisez des InitContainers pour les prérequis.
    • 
                    
  • Appliquez le principe du moindre privilège avec securityContext.
    • 
                    
  • Isoler les environnements avec des namespaces et Network Policies.
    • 
                    
  • Surveillez les logs et définissez des Resource Quotas.
    •
