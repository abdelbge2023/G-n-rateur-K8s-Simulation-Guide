📖 Introduction à Kubernetes

            

                Kubernetes (souvent abrégé K8s) est un orchestrateur de conteneurs open-source, initialement développé par Google. 
                Il permet d’automatiser le déploiement, la mise à l’échelle et la gestion d’applications conteneurisées.
            

            

                Objectifs de la formation :

                - Comprendre les cas d'utilisation des Pods.

                - Appliquer des exemples d'architectures.

                - Définir le rôle des services.

                - Maîtriser les volumes et le stockage persistant.
            


            
            
🌐 Comment fonctionne le réseau dans Kubernetes

            
    
                
  • Chaque Pod reçoit une adresse IP unique (et non chaque conteneur). Les Pods peuvent communiquer entre eux sans NAT.
    • 
                
  • Le kube-proxy gère les règles de réacheminement (iptables, IPVS) pour distribuer le trafic vers les Pods.
    • 
                
  • Le CNI (Container Network Interface) attribue les adresses IP et gère la connectivité entre les nœuds.
    • 
                
  • Les Services créent une abstraction stable (ClusterIP, NodePort, LoadBalancer).
    • 
                
  • L’Ingress apporte une couche de routage HTTP/HTTPS supplémentaire.
    • 
            

            
Résumé du flux réseau : Utilisateur → LoadBalancer (externe) → Ingress (optionnel) → Service (ClusterIP) → Pod.


            
            
🧩 Les Pods et leurs caractéristiques

            

                Qu'est-ce qu'un Pod ?

                Un Pod est le plus petit objet déployable dans Kubernetes. C'est un groupe d'un ou plusieurs conteneurs qui partagent :
                
    
                    
  • Un contexte réseau (une seule adresse IP).
    • 
                    
  • Un espace de noms Linux (network, UTS, IPC, PID).
    • 
                    
  • Des cgroups pour limiter, isoler et mesurer l'utilisation des ressources.
    • 
                    
  • Des volumes partagés (stockage).
    • 
                

            

            
Deux modèles d'utilisation :

            
    
                
  • Pod avec un seul conteneur : le cas le plus courant. Le Pod est une « enveloppe » autour du conteneur.
    • 
                
  • Pod avec plusieurs conteneurs : les conteneurs doivent travailler ensemble et partager des ressources (ex: serveur LAMP, groupe DNS/NTP/DHCP).
    • 
            


            
            
🔄 Cycle de vie des Pods

            
L'objet PodStatus (champ status) reflète l'état du Pod. Voici les différentes phases :

            

                

                    
ValeurDescription

                    
                        
PendingLe Pod a été accepté par Kubernetes, mais un ou plusieurs conteneurs ne sont pas totalement créés (téléchargement des images, configuration réseau...).

                        
RunningLe Pod a été affecté à un nœud, tous les conteneurs ont été créés. Au moins un conteneur est en cours d'exécution.

                        
SucceededTous les conteneurs du Pod ont terminé avec succès et ne seront pas redémarrés (cas des Jobs).

                        
FailedTous les conteneurs ont terminé, et au moins un a terminé en échec (code non nul ou arrêté par le système).

                        
UnknownL'état du Pod ne peut pas être obtenu (généralement en cas d'erreur de communication avec l'hôte).

                        
TerminatingLe Pod est en train de terminer son exécution (grace period).

                    
                

            

            
Consulter le cycle de vie :

            
# Exporter en JSON/YAML
kubectl get pod -n <namespace> <nom-pod> -o=jsonpath='{.status.conditions}' | jq
# Variables disponibles : lastProbeTime, lastTransitionTime, message, raison, Status


            
            
🩺 Les sondes (Probes)

            
Le kubelet contrôle l'état des conteneurs via des handlers. Trois types de sondes :

            
    
                
  • ExecAction : exécute une commande dans le conteneur. Succès si code retour = 0.
    • 
                
  • TCPSocketAction : teste la connexion sur l'IP:Port du conteneur.
    • 
                
  • HTTPGetAction : effectue une requête HTTP. Succès si statut 200.
    • 
            

            
Réponse du kubelet : Success, Failure, Unknown.

            
Les trois sondes principales :

            
    
                
  • livenessProbe : vérifie si le conteneur est vivant. Échec → redémarrage.
    • 
                
  • readinessProbe : vérifie si le conteneur est prêt à recevoir du trafic. Échec → retrait des endpoints.
    • 
                
  • startupProbe : vérifie le démarrage. Désactive les autres sondes jusqu'à succès.
    • 
            

            

                
Exemple de livenessProbe avec exec

                
apiVersion: v1
kind: Pod
metadata:
  name: pod-avec-probe
spec:
  containers:
  - name: app
    image: nginx
    livenessProbe:
      exec:
        command:
        - cat
        - /tmp/healthy
      initialDelaySeconds: 5
      periodSeconds: 5

                💡 Le Pod sera redémarré si le fichier /tmp/healthy n'existe pas.
            


            
            
⚙️ Les InitContainers – Exemple concret avec 04a_initContainer.yml

            

                Définition :

                Un InitContainer est un conteneur qui s'exécute avant les conteneurs principaux du Pod. Il est conçu pour réaliser des tâches d'initialisation (attendre qu'un service soit prêt, télécharger des données, configurer l'environnement).

                Caractéristiques :
                
    
                    
  • Ils s'exécutent en série (le suivant attend la fin du précédent).
    • 
                    
  • Ils doivent se terminer avec succès pour que le Pod passe à l'étape suivante.
    • 
                    
  • En cas d'échec, le kubelet les redémarre selon restartPolicy.
    • 
                    
  • Le statut est consultable via .status.initContainerStatuses.
    • 
                

            

            

                
📄 Fichier 04a_initContainer.yml – Pod avec 2 InitContainers

                
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  labels:
    app.kubernetes.io/name: MyApp
spec:
  containers:
  - name: myapp-container
    image: busybox:1.28
    command: ['sh', '-c', 'echo The app is running! && sleep 3600']
  initContainers:
  - name: init-myservice
    image: busybox:1.28
    command: ['sh', '-c', "until nslookup myservice.$(cat /var/run/secrets/kubernetes.io/serviceaccount/namespace).svc.cluster.local; do echo waiting for myservice; sleep 2; done"]
  - name: init-mydb
    image: busybox:1.28
    command: ['sh', '-c', "until nslookup mydb.$(cat /var/run/secrets/kubernetes.io/serviceaccount/namespace).svc.cluster.local; do echo waiting for mydb; sleep 2; done"]

                💡 Les InitContainers utilisent nslookup pour résoudre les services DNS. La commande inclut le namespace courant grâce à $(cat /var/run/secrets/kubernetes.io/serviceaccount/namespace).
                
🔍 Étapes de déblocage :

                
# 1. Déployer le Pod
kubectl apply -f 04a_initContainer.yml

# 2. Vérifier les logs du premier InitContainer
kubectl logs myapp-pod -c init-myservice
# Sortie : waiting for myservice - nslookup: can't resolve...

# 3. Créer le service myservice
apiVersion: v1
kind: Service
metadata:
  name: myservice
spec:
  ports:
  - protocol: TCP
    port: 80
    targetPort: 9376
# kubectl apply -f myservice.yml

# 4. Vérifier que le premier InitContainer est débloqué
kubectl logs myapp-pod -c init-myservice
# Sortie : Address 1: 10.109.138.31 myservice.kubernetes-tp.svc.cluster.local

# 5. Créer le service mydb (même principe)

# 6. Vérifier que le conteneur principal démarre
kubectl logs myapp-pod -c myapp-container
# Sortie : The app is running!

                💡 Les InitContainers sont bloquants : le conteneur principal ne démarre que lorsque tous les InitContainers ont réussi.
            


            
            
🛠️ Les SideCar Containers – Exemples concrets

            

                Définition :

                Un SideCar est un conteneur secondaire qui s'exécute aux côtés du conteneur principal dans le même Pod. Il étend ses fonctionnalités sans en altérer le fonctionnement.

                Cas d'usage : journalisation (log shipping), monitoring, proxy HTTPS, synchronisation de données, sécurité.

                Caractéristiques :
                
    
                    
  • Depuis Kubernetes 1.29, support natif via restartPolicy: Always sur un InitContainer.
    • 
                    
  • Ils démarrent séquentiellement mais continuent de tourner après le démarrage du Pod.
    • 
                    
  • Ils peuvent avoir leur propre readinessProbe affectant l'état global du Pod.
    • 
                

            


            
            

                
📄 04b_SideCar01.yml – Log Shipper avec Deployment

                
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: alpine:latest
          command: ['sh', '-c', 'while true; do echo "logging "`date` >> /opt/logs.txt; sleep 5; done']
          volumeMounts:
            - name: data
              mountPath: /opt
      initContainers:
        - name: logshipper
          image: alpine:latest
          restartPolicy: Always   # 👈 C'est ce qui en fait un SideCar !
          command: ['sh', '-c', 'tail -F /opt/logs.txt; echo "???"'] # -F = --follow=name --retry
          volumeMounts:
            - name: data
              mountPath: /opt
      volumes:
        - name: data
          emptyDir: {}

                💡 Le conteneur logshipper (SideCar) utilise restartPolicy: Always pour rester actif en permanence. Il lit en continu (tail -F) les logs écrits par myapp via le volume partagé emptyDir.
                
🖥️ Test :

                
kubectl apply -f 04b_SideCar01.yml
# Vérifier que le SideCar redémarre si la commande échoue
kubectl get pods
kubectl logs myapp-xxx -c logshipper   # Affiche les logs en temps réel

            


            
            

                
📄 04b_SideCar02.yml – Log Shipper avec un Job (tâche unique)

                
apiVersion: batch/v1
kind: Job
metadata:
  name: myjob
spec:
  template:
    spec:
      containers:
        - name: myjob
          image: alpine:latest
          command: ['sh', '-c', 'echo "logging" > /opt/logs.txt']
          volumeMounts:
            - name: data
              mountPath: /opt
      initContainers:
        - name: logshipper
          image: alpine:latest
          restartPolicy: Always
          command: ['sh', '-c', 'tail -F /opt/logs.txt']
          volumeMounts:
            - name: data
              mountPath: /opt
      restartPolicy: Never
      volumes:
        - name: data
          emptyDir: {}

                💡 Ici, le SideCar logshipper suit les logs du conteneur principal (myjob) qui écrit une seule fois. Le SideCar reste actif pour permettre la récupération des logs avant que le Job ne se termine.

📄 Étape 2 : Génération des certificats TLS et création du Secret

                
# Génération d'une clé privée et d'un certificat auto-signé (valable 365 jours)
openssl req -newkey rsa:2048 -nodes -keyout nginx.key -x509 -days 365 -out nginx.crt

# Création du Secret dans Kubernetes (stockage sécurisé)
kubectl create secret generic nginx-certs-keys --from-file=./nginx.crt --from-file=./nginx.key

# Vérification du Secret
kubectl describe secrets/nginx-certs-keys

                💡 Les Secrets sont utilisés pour stocker des données sensibles (certificats, clés privées).

                
                
📄 Étape 3 : Fichier de configuration Nginx (04c_nginx_proxy_02_nginx_default.conf)

                
server {
    listen 443 ssl;
    root /usr/share/nginx/html;
    index index.html;
    server_name localhost;
    ssl_certificate /etc/nginx/ssl/nginx.crt;
    ssl_certificate_key /etc/nginx/ssl/nginx.key;
    ssl_session_timeout 1d;
    ssl_session_cache shared:SSL:50m;
    ssl_session_tickets off;
    ssl_protocols TLSv1.2;
    ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:...';
    ssl_prefer_server_ciphers on;
    add_header Strict-Transport-Security max-age=15768000;
    ssl_stapling on;
    ssl_stapling_verify on;
    location / {
        proxy_pass http://localhost/;   # 👈 Proxy vers Apache sur localhost:80
    }
}

                💡 Configuration robuste : TLSv1.2, HSTS, OCSP Stapling. Le proxy redirige le trafic HTTPS vers Apache en HTTP sur localhost:80.

                
                
📄 Étape 4 : Création du ConfigMap

                
kubectl create configmap nginxconfigmap --from-file=default.conf=04c_nginx_proxy_02_nginx_default.conf


                
                
📄 Étape 5 : Déploiement Apache + Nginx SideCar – version corrigée

                
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-proxy
  namespace: kubernetes-tp
  labels:
    app: nginx-proxy
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx-proxy
  template:
    metadata:
      labels:
        app: nginx-proxy
    spec:
      volumes:
      - name: secret-volume
        secret:
          secretName: nginx-certs-keys
      - name: configmap-volume
        configMap:
          name: nginxconfigmap
      containers:
      - name: apache
        image: ubuntu/apache2
        ports:
        - containerPort: 80
      initContainers:
      - name: nginxhttps
        restartPolicy: Always
        image: nginx:alpine                # ✅ Image compatible containerd 2.1
        command: ["nginx", "-g", "daemon off;"]   # ✅ Démarrage Nginx en avant‑plan
        ports:
        - containerPort: 443
        volumeMounts:
        - mountPath: /etc/nginx/ssl
          name: secret-volume
        - mountPath: /etc/nginx/conf.d
          name: configmap-volume
  minReadySeconds: 30
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0

                💡 Points clés :

                - restartPolicy: Always sur l'InitContainer → SideCar.

                - Image officielle nginx:alpine.

                - Commande nginx -g "daemon off;" pour rester attaché.

                - Les volumes montent le Secret (certificats) et le ConfigMap (configuration).

📦 Volumes de stockage

            
Nouveau Cette section reprend l’intégralité du chapitre sur les volumes de la formation.


            
📌 Problématique du stockage par défaut

            
Par défaut, le stockage dans les conteneurs est :

            
    
                
  • Exclusif à chaque conteneur (non partagé).
    • 
                
  • Éphémère : perdu à chaque redémarrage ou suppression du Pod.
    • 
                
  • Lancé avec un état « propre » (clean state) – rien n’est conservé.
    • 
            

            

                Solution : Kubernetes propose l’abstraction Volume pour :
                
    
                    
  • Persister des données (configuration, clés, données applicatives).
    • 
                    
  • Partager des fichiers entre conteneurs, Pods ou même avec l’extérieur.
    • 
                    
  • Offrir des stockages temporaires (quand on en a besoin).
    • 
                

            


            
📄 Déclaration d’un volume dans un Pod

            
Deux sections dans le YAML :

            
    
                
  • .spec.volumes – on définit le volume (nom, type, source).
    • 
                
  • .spec.containers[*].volumeMounts – on déclare le point de montage dans chaque conteneur.
    • 
            

            

                
Exemple extrait du fichier 04b_SideCar01.yml

                
volumes:
- name: data
  emptyDir: {}        # Volume temporaire partagé

containers:
- name: myapp
  volumeMounts:
  - name: data
    mountPath: /opt   # Monté dans /opt du conteneur

                💡 On voit ici un volume emptyDir partagé entre les conteneurs du même Pod.
            


            
🧩 Types de volumes principaux

            
    
                
  • emptyDir : volume temporaire créé au démarrage du Pod, vide, détruit avec le Pod. Utile pour le partage entre conteneurs.
    • 
                
  • hostPath : monte un répertoire ou fichier du nœud (hôte) dans le conteneur. À utiliser avec précaution (problèmes de portabilité Windows/Linux).
    • 
                
  • configMap (lecture seule) : injecte des fichiers de configuration depuis un ConfigMap.
    • 
                
  • secret : injecte des données sensibles (certificats, tokens) dans un volume temporaire (tmpfs).
    • 
                
  • persistentVolumeClaim : utilise un PersistentVolume (PV) pour un stockage persistant (voir ci‑dessous).
    • 
                
  • projected : agrège plusieurs sources (secret, configMap, downwardAPI, serviceAccountToken) dans un seul répertoire.
    • 
            


            
🔌 Container Storage Interface (CSI)

            

                CSI est une norme qui permet aux fournisseurs de stockage (cloud, CEPH, NFS, etc.) de proposer leur solution à Kubernetes via une interface unifiée. Chaque fournisseur implémente les drivers CSI pour exposer son stockage.
            


            
            
🧠 Comprendre PV, PVC et Pod – L’essentiel du stockage persistant

            

                Pour bien comprendre le stockage dans Kubernetes, il faut distinguer trois notions fondamentales : le Pod (l’application), le PersistentVolume (PV) (la ressource de stockage) et le PersistentVolumeClaim (PVC) (la demande de stockage). Ces trois éléments forment la chaîne d’approvisionnement du stockage persistant.
            


            

                📌 Analogie simple : Imaginez que Kubernetes est un déménageur.
                
    
                    
  • Le PV est le camion de déménagement (la ressource – capacité 100 m³).
    • 
                    
  • Le PVC est votre bon de commande (je veux 30 m³ de ce camion).
    • 
                    
  • Le Pod est votre nouvel appartement (l’application) qui contient le salon (le dossier /usr/share/nginx/html).
    • 
                    
  • Le volumeMounts dans le Pod est l’ordre donné au déménageur : « Décharge les 30 m³ du camion directement dans le salon ! »
    • 
                

            


            
1. Le Pod – Le « Locataire » / L’application

            
    
                
  • Définition : Unité la plus petite déployable. Un ou plusieurs conteneurs partagent réseau, IPC, volumes.
    • 
                
  • Rôle : Héberge votre application (ex: serveur Nginx).
    • 
                
  • Caractère : Éphémère – s’il crash, il est recréé et les données internes sont perdues (à moins d’utiliser un volume persistant).
    • 
                
  • Dans le TP : task-pv-pod est le Pod qui contient Nginx. Il monte le volume via volumeMounts pour servir la page HTML.
    • 
            


            
2. Le PersistentVolume (PV) – Le « Disque dur »

            
    
                
  • Définition : Ressource de stockage provisionnée par l’administrateur, indépendante des Pods.
    • 
                
  • Rôle : Représente un morceau de stockage (disque local, NFS, volume cloud).
    • 
                
  • Caractéristiques : Non‑namespacé (visible de tout le cluster), persistant, cycle de vie indépendant.
    • 
                
  • Dans le TP : task-pv-volume est un PV hostPath pointant vers /mnt/data sur l’hôte.
    • 
            


            
3. Le PersistentVolumeClaim (PVC) – Le « Bon de commande »

            
    
                
  • Définition : Demande de stockage effectuée par un utilisateur (un développeur).
    • 
                
  • Rôle : Spécifie la taille, les droits d’accès et la classe de stockage souhaités.
    • 
                
  • Caractéristiques : Namespacé (appartient à un namespace), recherche automatiquement un PV correspondant et se lie (bind) à lui.
    • 
                
  • Dans le TP : task-pv-claim demande 30Mi en RWO. Kubernetes le lie au PV task-pv-volume.
    • 
            


            
🔗 Comment ils interagissent (l’enchaînement logique)

            

                
    
                    
  1. L’administrateur crée le PV (ex: task-pv-volume de 100Mi).
    2. 
                    
  2. Le développeur crée le PVC (ex: task-pv-claim demandant 30Mi).
    3. 
                    
  3. Kubernetes lie (bind) le PVC au PV correspondant.
    4. 
                    
  4. Le développeur crée le Pod en spécifiant le PVC dans son volume.
    5. 
                    
  5. Le Pod démarre et voit le volume monté dans son système de fichiers.
    6.

💡 Pourquoi cette séparation est géniale

            
    
                
  • Découplage : Le développeur ne se soucie pas du type de stockage (local, NFS, cloud). Il utilise juste un PVC.
    • 
                
  • Centralisation : L’administrateur gère les PV, peut les migrer sans impacter les développeurs.
    • 
                
  • Réutilisation : Un PV peut être partagé par plusieurs Pods (en lecture seule par exemple).
    • 
            


            
            
🔧 Exemple pratique avec hostPath (version simplifiée)

            
Cet exemple est une version simplifiée du TP (sans sshfs). Il utilise un répertoire local sur chaque nœud.

            

                
📄 Préparer le stockage sur les nœuds

                
sudo mkdir -p /mnt/data
sudo sh -c 'echo "Hello from Kubernetes storage" > /mnt/data/index.html'
sudo chmod 755 /mnt/data
sudo chmod 644 /mnt/data/index.html


                
📄 PV, PVC, Pod (YAML)

                
# PV
apiVersion: v1
kind: PersistentVolume
metadata:
  name: task-pv-volume
  labels:
    type: local
spec:
  storageClassName: manual
  capacity:
    storage: 100Mi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: "/mnt/data"
---
# PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:

🔧 TP PersistentVolumes avec sshfs – Correction complète

            

                Cette section reprend le TP des pages 28 à 31 du document, en utilisant sshfs et fstab pour monter un répertoire distant depuis le master vers les workers. Le document original indique « NON FONCTIONNEL », nous allons le corriger pas à pas.
            

            

                📌 Problèmes identifiés dans le TP original :
                
    
                    
  • Conflit entre fuse et fuse3 sur Debian Bookworm → nécessite de désinstaller fuse et d’installer fuse3.
    • 
                    
  • Absence de clé SSH → il faut générer et copier les clés.
    • 
                    
  • Le fichier index.html n’existe pas sur le nœud où le Pod est planifié (ici ker1).
    • 
                    
  • La ligne fstab est mal rédigée (manque allow_other, IdentityFile).
    • 
                    
  • Le montage n’est pas effectué après modification de fstab → il faut exécuter sudo mount /mnt/data.
    • 
                    
  • Droits insuffisants sur le répertoire monté → chmod 755 et 644.
    • 
                    
  • Le conteneur Nginx tourne avec un utilisateur non‑root → ajouter securityContext.runAsUser: 0.
    • 
                

            


            
📋 Procédure pas à pas

            
1. Générer et copier les clés SSH (sur le master)

            
# Générer une clé (sans phrase de passe)
ssh-keygen -t rsa -b 2048 -N "" -f ~/.ssh/id_rsa

# Copier vers chaque worker (et vers le master pour le montage local)
ssh-copy-id abdel@keb2
ssh-copy-id abdel@keb3
ssh-copy-id abdel@ker1


            
2. Créer le répertoire de données sur le master

            
mkdir -p /home/abdel/data
echo "Hello from Kubernetes storage" > /home/abdel/data/index.html
chmod 644 /home/abdel/data/index.html
chmod 755 /home/abdel/data


            
3. Installer sshfs sur les workers (sans conflit)

            
# Sur chaque worker (keb2, keb3)
sudo apt update
sudo apt remove fuse          # Si présent
sudo apt install -y fuse3 sshfs


            
4. Créer le point de montage sur les workers

            
sudo mkdir -p /mnt/data


            
5. Tester le montage manuel

            
sudo sshfs -o allow_other abdel@ker1:/home/abdel/data /mnt/data
ls -la /mnt/data/
cat /mnt/data/index.html      # Doit afficher le message
sudo fusermount -u /mnt/data   # Démonter pour passer à fstab


            
6. Configurer le montage automatique via /etc/fstab

            
# Sur chaque worker, éditer /etc/fstab :
sudo nano /etc/fstab
# Ajouter la ligne suivante (adaptez l'utilisateur) :
abdel@ker1:/home/abdel/data /mnt/data fuse.sshfs port=22,user,noauto,noatime,_netdev,allow_other,IdentityFile=/home/abdel/.ssh/id_rsa 0 0

# Monter le répertoire
sudo mount /mnt/data


            
7. Ajuster les droits

            
sudo chmod 755 /mnt/data
sudo chmod 644 /mnt/data/index.html


            
8. Créer les ressources Kubernetes (PV, PVC, Pod)

            
Utiliser les mêmes fichiers YAML que pour l’exemple hostPath (ils sont identiques, car le PV pointe vers /mnt/data).

            
# PV
apiVersion: v1
kind: PersistentVolume
metadata:
  name: task-pv-volume
  labels:
    type: local
spec:
  storageClassName: manual
  capacity:
    storage: 100Mi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: "/mnt/data"
---
# PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: task-pv-claim
spec:
  storageClassName: manual
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 30Mi
---
# Pod
apiVersion: v1
kind: Pod
metadata:
  name: task-pv-pod
spec:
  volumes:
    - name: task-pv-storage
      persistentVolumeClaim:
        claimName: task-pv-claim
  containers:
    - name: task-pv-container
      image: nginx
      ports:
        - containerPort: 80
      volumeMounts:
        - mountPath: "/usr/share/nginx/html"
          name: task-pv-storage
      securityContext:
        runAsUser: 0   # Pour éviter l'erreur 403


            
9. Déployer et tester

            
kubectl apply -f pv-hostpath.yml
kubectl apply -f pvc-claim.yml
kubectl apply -f pod-with-pvc.yml
kubectl get pods -o wide
curl http://<IP_DU_POD>
# Sortie : Hello from Kubernetes storage


            

                🔍 Vérifications supplémentaires :
                
    
                    
  • Assurez‑vous que le montage est actif sur le worker où le Pod est planifié : ls -la /mnt/data/
    • 
                    
  • Vérifiez les logs du Pod en cas d’erreur : kubectl describe pod task-pv-pod
    • 
                    
  • Pour nettoyer : kubectl delete pod task-pv-pod ; kubectl delete pvc task-pv-claim ; kubectl delete pv task-pv-volume
    •

Exemple 4 : Stockage persistant avec PersistentVolumeClaim

            

                
🎯 Objectif : Conserver des données même après redémarrage du Pod.

                
📄 PVC et Pod

                
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mon-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: pod-avec-stockage
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: stockage
      mountPath: /usr/share/nginx/html
  volumes:
  - name: stockage
    persistentVolumeClaim:
      claimName: mon-pvc

🔒 Sécurité & Administration (bonnes pratiques)

            

                Bonnes pratiques pour les Pods :
                
    
                    
  • Utilisez des sondes (livenessProbe, readinessProbe) pour la résilience.
    • 
                    
  • Utilisez des InitContainers pour les prérequis.
    • 
                    
  • Appliquez le principe du moindre privilège avec securityContext.
    • 
                    
  • Isoler les environnements avec des namespaces et Network Policies.
    • 
                    
  • Surveillez les logs et définissez des Resource Quotas.
    • 
                    
  • Pour les certificats, stockez-les dans des Secrets, jamais en clair.
    • 
                    
  • Pour les volumes persistants, utilisez des PVC et évitez hostPath en production.
